# Decisions

Running log of non-obvious technical/product decisions made while building Peguei, so later work doesn't contradict earlier reasoning.

## Identity

- **App name**: Peguei (`app.json` displayName, `strings.xml`, `Info.plist` CFBundleDisplayName).
- **Bundle ID / applicationId**: `com.peguei.app` on both platforms. No real domain was ever specified for this project, so this is the reasonable default (product name + `.app`), applied consistently to iOS `PRODUCT_BUNDLE_IDENTIFIER` and Android `applicationId`.
- **Native package/namespace**: Android Kotlin package and Gradle `namespace` stay `com.voicecambuffer` (the original scaffold name) rather than following `applicationId`. Renaming a Kotlin package after native modules already exist is high-risk churn (every file's `package` line, every import) for zero user-visible benefit — `applicationId` is what matters for Play Store identity and app-to-app collisions, not the internal Kotlin package. Same logic for the Xcode project/scheme name (`VoiceCamBuffer.xcodeproj`, target `VoiceCamBuffer`) and the npm package name in `package.json` — these are internal/dev-facing identifiers, not user-facing branding.
- **App icon**: anzol (fishhook) entwined with a camera shutter, teal/petroleum (`#0B4F4F`) + orange accent (`#F5A623`), flat style. Source SVGs in `assets/icon/`; rendered into the full iOS `AppIcon.appiconset` and Android adaptive icon (foreground/background per density).

## Wake word (removed — this section corrects earlier, now-stale entries)

**Everything previously written in this section describing a working Picovoice/Porcupine wake-word feature ("Peguei!"/"Fisgou!" presets, sensitivity slider, "Modo Pesca", `useWakeWord.ts`) no longer reflects the code.** Sometime between the audit that produced those entries and this "refinamento comercial/UX" pass, the entire feature was stripped from the repo: `package.json` has zero Picovoice dependencies, there's no `useWakeWord.ts` hook, no wake-word UI in `SettingsScreen`, no `assets/wakewords/`, no `patches/`, and no microphone permission anywhere (`Info.plist`/`AndroidManifest.xml`/`usePermissions.ts` request camera + storage only). The only remaining trace was a stale native `.cxx` build-cache reference to a `VoiceProcessor` module that no longer exists in source. This was discovered mid-pass (the task assumed wake word existed, matching the old docs) and confirmed with the user before proceeding: **ship as a manual-only app**; do not rebuild wake word as part of this pass. See `BLOCKED.md` if it's ever reintroduced — the old presets/asset-path plumbing described above would need to be rebuilt from scratch, not restored, since none of it exists anymore.

Practical effect on this pass: the onboarding slide originally planned around the wake word ("Diga 'Peguei!' e pronto") was replaced with a "tap the button" slide instead (see "Onboarding" below); the top-bar "Modo Pesca" toggle called for in the redesign brief was skipped entirely, since there's no feature left for it to control; and the privacy policy / ASO copy make no voice-command or on-device-voice-processing claims.

## Buffer/save architecture (verified, not a bug)

- Audited the "core value prop" concern (does tapping/wake-word only trigger the *cut*, never the *start*, and does the saved clip include pre-roll) end-to-end on both platforms. It's correct as built: `CameraScreen` starts the native buffer unconditionally on mount (independent of any button), and `CameraEncoderController.saveClip()` (Android Kotlin, iOS Swift) takes `ringBuffer.snapshotFromOldestKeyframe()` as the clip's head *before* appending any new frames, then keeps appending live frames as a tail until `postRollSeconds` have elapsed past the tap. No fix was needed.
- Added `SaveClipDurationTest.kt` / `SaveClipDurationTests.swift` (mirrored, same pattern as the existing `FrameRingBufferTest`/`FrameRingBufferTests` pair) to make this regression-proof. `CameraEncoderController` itself can't run under plain JUnit/XCTest (it drives Camera2/MediaCodec/MediaMuxer or AVFoundation/VideoToolbox, which need real hardware), so the tests instead replicate its exact head+tail assembly algorithm against a synthetic `FrameRingBuffer` frame stream and assert the resulting duration is `elapsedPreRoll + postRoll`, explicitly checking it's *not* just `postRoll` alone (the regression this whole exercise was worried about). This is the same "test the pure algorithm, not the hardware-bound wrapper" pattern already established by `FrameRingBufferTest`.

## Real bug found on first physical-device run: buffer never actually started (iOS)

- **Symptom:** camera preview stayed black and `saveClip()` always failed with "O buffer não está ativo," discovered running the app on a physical iPhone for the first time (this exact code path had never run against real camera hardware before — prior verification was static review + pure-algorithm unit tests only).
- **Root cause:** `CameraScreen` passed `isActive={phase === 'buffering'}` to `<CircularBufferPreview>`. At mount, `phase` is `'idle'`, so the view's `isActive` prop starts `false`. `CircularBufferPreviewView.swift`'s `isActive` `didSet` reacts to that by calling `CameraEncoderController.shared.detachPreviewLayer()`, which calls `stop()` — an **async** block (dispatched on `sessionQueue`) that unconditionally wipes `pendingConfig` and `ringBuffer` to `nil`. Meanwhile, the JS mount effect calls `start()` → native `configureAndArm()`, which sets `pendingConfig` (synchronously, on whatever thread the bridge calls it on). If the async `stop()` block from the initial `isActive=false` transition executes *after* `configureAndArm()` set `pendingConfig`, it silently destroys that state. When `phase` later becomes `'buffering'` and `isActive` flips to `true`, `attachPreviewLayer()` checks `pendingConfig != nil` to decide whether to call `openCameraAndStart()` — but by then it's `nil`, so the camera session is **never opened**. `isRunning` stays `false` forever; the native `startBuffering()` promise had already resolved unconditionally (see the "Native error event plumbing" section above), so the JS side has no way to know this happened — it just shows `phase: 'buffering'` with a black preview until the user taps save and hits the `NOT_BUFFERING` error path.
- **Compounding design issue, not just a race:** tying `isActive` to `phase === 'buffering'` was wrong regardless of the race — `phase` also transitions away from `'buffering'` during every `'saving'` (post-roll capture) and any `'error'`, which would have torn down the *entire* capture session mid-use even without the startup race, directly contradicting "the buffer must stay continuously active for as long as the camera screen is open."
- **Fix:** `CameraScreen` now passes `isActive` unconditionally (`<CircularBufferPreview ... isActive />`), matching the actual requirement — the preview/session should be active for the screen's entire mounted lifetime, independent of the specific sub-phase. This also means the prop's value at mount now equals `CircularBufferPreviewView`'s own Swift default (`true`), so the destructive `didSet` branch never fires during normal use at all; real teardown still happens correctly via `didMoveToWindow()` when the view actually unmounts.
- **Not touched:** the Android side (`CircularBufferPreviewView.kt`) doesn't have this bug — its `isActive` flag only gates whether a *newly-available* `SurfaceTexture` auto-attaches; it never reactively tears down an already-running session when the flag flips, and `onSurfaceTextureDestroyed` (the actual teardown trigger) is tied to the real surface lifecycle, not to the JS-controlled prop. No change was needed there, but this is worth knowing if the same `isActive`-driven reactive teardown pattern is ever copied to Android in the future.

## Recording model changed: manual start/stop replaces the fixed post-roll (2026-07-11)

**This reverses part of the original spec** ("the button only ever triggers a save, never a start/stop") **per explicit user direction after testing the always-buffering + fixed-post-roll version on a physical device.** The buffer itself is unchanged (still always recording pre-roll from the moment the camera screen opens, independent of any button) — what changed is what happens when the user taps the capture button:

- **Before:** one tap saved a clip immediately: buffered pre-roll (`bufferSeconds`) + a *fixed* configured post-roll (`postRollSeconds`), no further user input.
- **Now:** tapping starts an open-ended manual recording (no fixed duration); tapping again stops it. The saved clip concatenates the pre-roll buffer snapshot (taken the moment recording started) with whatever was recorded between the two taps — the "post-roll" length is now however long the user chooses to keep recording, not a setting.
- **`postRollSeconds` setting was removed entirely** (`POST_ROLL_SECONDS_MIN/MAX`, the Settings slider, `AppSettings.postRollSeconds`) — it has no meaning once post-roll length is user-controlled rather than a fixed timer. `bufferSeconds` (pre-roll length) is unaffected and still configurable.
- **Native API changed on both platforms:** `CameraEncoderController.saveClip(postRollSeconds, ...)` was split into `startManualRecording(...)` (snapshots the pre-roll head, arms indefinite tail accumulation) and `stopManualRecording(...)` (finalizes using whatever tail frames accumulated, no target timestamp check). `CircularBufferModule.saveClip` was replaced by `startManualRecording`/`stopManualRecording` (both platforms' native module + Obj-C bridge for iOS). JS: `useCircularBuffer.saveClip()` was replaced by `startRecording()`/`stopRecording()`; `RecordingPhase` gained a `'recording'` state (open-ended, between the two taps) distinct from `'saving'` (the brief mux/finalize step right after stop).
- **`RecordButton` reworked accordingly:** the circular post-roll progress ring (which visualized "% of the way to a known target duration") no longer makes sense with an open-ended duration, so it was replaced with a pulsing recording indicator + a live elapsed-time readout (`0:07`, etc.) — standard camera-app feedback for "how long have I been recording," not a violation of the "no text labels" rule (that rule was about instructional labels like "Salvar clipe", not live numeric feedback).
- **Tests updated to match:** `SaveClipDurationTest.kt`/`SaveClipDurationTests.swift` now simulate "start, accumulate N seconds of tail frames, stop" instead of "start with a fixed target timestamp" — still proving the core guarantee (saved duration = elapsed pre-roll + recording length, never just the recording length alone), plus a new case proving a longer manual recording isn't capped at any fixed duration.
- **Not yet updated:** onboarding copy (slide 2, "Toque o botão. Só isso.") and the ASO/store-listing copy still describe the single-tap fixed-post-roll flow from the original design pass, since they were written before this change. These need a follow-up pass to describe start/stop instead — flagging this explicitly so it isn't missed.

## Real bug found on first physical-device run: navigating away from Camera (e.g. to Settings) permanently killed the buffer (iOS)

- **Symptom:** after the `isActive`-always-true fix above, the buffer started correctly on launch, but navigating to Settings or Gallery and back left the camera preview black forever, with `saveClip`/`startManualRecording` failing with "O buffer não está ativo."
- **Root cause:** `react-native-screens` detaches an unfocused screen's native view from the `UIWindow` for performance (confirmed via added `NSLog` tracing: `didMoveToWindow` fired with `window=false` when navigating away, and `window=true` again on return — this is **not** a `CameraScreen` unmount, the JS component and its state stay alive the whole time). `CircularBufferPreviewView.didMoveToWindow` responded to `window == nil` by calling `detachPreviewLayer()`, which called the **full** `stop()` — wiping `pendingConfig` along with everything else. When the view reattached, `attachPreviewLayer()`'s restart check (`pendingConfig != nil && !isRunning`) failed because `pendingConfig` was gone, so the camera never restarted.
- **Android didn't have this bug**, and comparing the two platforms is what revealed the fix: `CircularBufferPreviewView.kt`'s surface-destroyed callback calls `CameraEncoderController.stopInternal()` directly, **not** the public `stop()` — and only `stop()` (used by the real, explicit `CircularBufferModule.stopBuffering()`) clears `pendingConfig`. Android's split already existed; iOS's didn't.
- **Fix:** split iOS's `stop()` the same way — `stopInternal(clearPendingConfig:)` handles the actual session/encoder teardown, and only the public `stop()` (called from `CircularBufferModule.stopBuffering()`, i.e. a real explicit stop) passes `clearPendingConfig: true`. `detachPreviewLayer()` now calls `stopInternal(clearPendingConfig: false)`, so a screen-focus-driven detach/reattach cycle correctly re-arms and restarts the camera instead of leaving it dead.
- **Known remaining gap, not fixed in this pass:** this fix stops the camera hardware/encoder while the preview is detached (matching Android) and correctly restarts it on return — it does **not** keep the buffer recording video in the background while the user is on Settings/Gallery with Camera off-screen. True uninterrupted background buffering across screen navigation (as opposed to "correctly restarts every time you come back") would require not tearing down the `AVCaptureSession` on detach at all, only detaching the visual preview layer — a larger, separate change not attempted here given the immediate bug was the dead-forever case, not the brief gap while navigating.
- **Debug instrumentation:** temporary `NSLog("[Peguei][debug] ...")` tracing was added throughout `CameraEncoderController.swift`/`CircularBufferModule.swift`/`CircularBufferPreviewView.swift` to diagnose this live against the physical device; remove it once the fix is confirmed stable (search for `[Peguei][debug]`).

## Real crash found on physical device: saving to Photos crashed the whole app (iOS) -- feature reverted, not fixed

- **Symptom:** the app hard-crashed (SIGABRT) every single time immediately after a clip finished saving. Live `NSLog` checkpoint tracing throughout `finalizeSave` proved the native muxing itself always completed successfully (`writer.status == .completed`) -- the crash happened in whatever ran right after, which is exactly where the "also save to Photos" call (`saveClipToDeviceGallery`, via `@react-native-camera-roll/camera-roll`) fired.
- **What was tried, in order, none of which fixed it:**
  1. Hardening `finalizeSave`'s `AVAssetWriterInput.append(_:)` loop against writer-status races (worth keeping regardless -- real defense-in-depth against a different theoretical crash -- but not the cause here).
  2. Removing the custom `album` option from the `CameraRoll.saveAsset()` call, on the theory that `PHAssetCollection fetchAssetCollectionsWithType:` (needed to find/create a named album) requires broader `PHAccessLevelReadWrite`/`NSPhotoLibraryUsageDescription` access than this app declares (`NSPhotoLibraryAddUsageDescription` only) -- plausible from reading `RNCCameraRoll.mm`, but the crash persisted identically without an album, so this wasn't the actual trigger (or wasn't the only one).
  3. Dispatching the native module's `resolve()`/`reject()` calls onto the main thread (`DispatchQueue.main.async`) before calling them, since `CameraEncoderController`'s callbacks fire from a background `sessionQueue`/`DispatchQueue.global`, and resolving a TurboModule promise off the main thread is a known crash source with the New Architecture (enabled in this project). Confirmed via JS-side `console.log` tracing that **zero JS logs ever fired** -- the crash happens before the resolved promise value ever reaches JS, i.e. natively, inside or around the bridge resolution itself. Still didn't fix it.
- **Actual resolution: the "save to system Photos" feature was removed entirely**, per explicit user direction after confirming (by removing the feature) that this was indeed the crash source. `saveClipToDeviceGallery()`/`src/utils/cameraRoll.ts` and the `@react-native-camera-roll/camera-roll` dependency were deleted; `useCircularBuffer.stopRecording()` no longer calls anything after `addClip()`/`setPhase('buffering')`. Clips are saved **only** to the app's own sandboxed storage again, same as the original design -- there is no "save to system gallery" feature in this app. The exact underlying crash mechanism inside the camera-roll library/bridge interaction was never fully root-caused at the disassembly level (lldb attach to the physical device could not be established through this project's ad-hoc/personal-team signing setup via the command line -- `device process attach` consistently got stuck in `state=attaching`, most likely a missing debug entitlement that Xcode's own "Run" workflow injects automatically but a raw `xcodebuild` invocation does not).
- **Product implication:** if "save to system Photos/Gallery" is wanted again in the future, it needs to be revisited as its own careful piece of work (ideally with proper Xcode-attached debugging available, not just device console logs), not re-added as a quick addition. The in-app Gallery is the sole place clips live for now; see the next section for its planned redesign.
- **Debug instrumentation cleanup:** all temporary `[Peguei][debug]`/`[Peguei][debug-js]` `NSLog`/`console.log` tracing added during this investigation has been removed from `CameraEncoderController.swift`, `CircularBufferModule.swift`, `CircularBufferPreviewView.swift`, and `useCircularBuffer.ts` now that the crash is confirmed resolved. The `DispatchQueue.main.async` wrapping around `resolve`/`reject` in `startManualRecording`/`stopManualRecording` was kept as a real hardening improvement even though it wasn't the fix for this specific crash.

## Real bug found on physical device: gallery/clip thumbnails always failed on iOS

- **Symptom:** `getClipThumbnail()` always returned null on iOS (falling back silently to the placeholder icon) with a device-log error "The file couldn't be opened because you don't have permission to view it" from `react-native-create-thumbnail`.
- **Root cause:** the library's two native implementations disagree on input format. Android's explicitly checks for a `file://` prefix and strips it before use; iOS's calls `NSURL(fileURLWithPath:)` directly on anything that isn't `http(s)://`, so handing it an already-`file://`-prefixed string makes it treat the literal text `"file://..."` as a path component to resolve relative to something else, which never exists -- surfacing as a generic permission error rather than a clear "not found."
- **Fix:** `getClipThumbnail()` (`src/utils/thumbnails.ts`) now strips any `file://` prefix from the input and only re-adds it on Android (`Platform.OS === 'android'`), passing the bare path on iOS.

## Capture button and camera top-bar redesign

- **`RecordButton` redesigned to icon-only states** (record circle → stop-square with a circular post-roll progress ring → brief checkmark flash on save), no text label, per the camera-app convention (iOS Camera, Snapchat, Instagram). Progress ring duration is driven by an `Animated.timing` tied to `postRollSeconds`, not by a real byte-level sync with the native save — it's a UI approximation of "how much of the post-roll window has elapsed," which is what "borda de progresso circular sutil" calls for.
- **Added `react-native-svg`** to draw the progress ring, gear icon, gallery placeholder icon, and onboarding illustrations. No custom icon/vector library existed before; this is the standard, low-risk choice (autolinked, no manual native code, verified building clean on both `pod install` and Gradle).
- **`RecordButton` was previously double-purposed** as a generic "grant permission" button on `PermissionsScreen` (same component, `saving={false}` misused as a way to render a plain button). Since it's now a capture-specific icon control, that reuse was split out into a new `PermissionActionButton` (simple labeled pill button). `PermissionsScreen.test.tsx` was updated accordingly (`testID` changed from the shared `record-button` to `permission-action-<key>`).
- **Gallery button (bottom-left) shows a real video-frame thumbnail** of the last saved clip, not a generic icon, per the redesign brief. Added **`react-native-create-thumbnail`** (small, single-purpose, well-established native module) to extract a frame; `ClipThumbnail` in the Gallery list itself was also upgraded to use it for consistency. Falls back to a neutral placeholder icon (dimmed opacity) when there's no clip yet or thumbnail extraction fails.
- **Settings gear (top-right) and the top-left buffer-status badge use a semi-transparent circular "scrim" (`rgba(10,20,20,0.55)`), not an actual blur view.** The brief calls the scrim (not the blur) the primary WCAG AA contrast mechanism ("usando o scrim semi-transparente como estratégia principal"); a real blur would need a new native dependency (`@react-native-community/blur` or similar) with known Android finickiness, for a purely decorative layer on top of a mechanism that already satisfies the actual contrast requirement on its own. Deferred rather than added — see `ScrimIconButton`.
- Modo Pesca toggle: not implemented in the redesigned top bar — see "Wake word (removed)" above.

## Brand identity (`BRAND.md`, `src/theme/`)

- Palette, tone-of-voice, and a type scale are documented in `BRAND.md` and implemented as tokens in `src/theme/colors.ts` / `src/theme/typography.ts`, applied across every screen and shared component (no more ad hoc hex literals per screen).
- **Typography renders on the OS system font, not real bundled Inter `.ttf` files.** The project has no existing custom-font pipeline (no `assets/fonts/`, no `Info.plist` `UIAppFonts` entry, no Android font-asset wiring); standing that up is a separate, self-contained piece of native work disproportionate to this pass. `BRAND.md` documents exactly what's needed to swap in real Inter later.
- **No runtime light/dark theme switcher was added**, even though light-mode token values are defined in `colors.ts`. Every screen in this app already reads as dark-themed by design (the camera preview is inherently black; Settings/Gallery/Permissions already used near-black backgrounds before this pass) and none of them currently opt into light mode — building a `useColorScheme`-driven theme provider is a separate scope increase from "define a palette that includes light values for later," which is what the brief actually asked for.

## Clip metadata (corrected — the entry below described code that doesn't exist)

**This section previously described a `clip_<manual|voice>_<epochMillis>.mp4` filename scheme and a `triggeredBy: 'voice' | 'manual'` field.** Neither exists in the code as of this pass: the native save path (`CameraEncoderController` on both platforms) has only ever written `clip_<epochMillis>.mp4`, with no origin marker, and `SavedClip` had no `triggeredBy` field at all until the gallery rebuild below added one. This was presumably written when wake word still existed and never actually implemented, then went stale when wake word was removed (see "Wake word (removed)" above) — worth knowing so future entries aren't trusted at face value without checking the code.

- **`triggeredBy` is now a real field (`ClipTrigger = 'manual'`, `src/types/index.ts`), but it's a single-value type, not a union with 'voice'.** There's no second capture trigger in this app, so it's always `'manual'`. It's still modeled as an explicit field (not just assumed) and shown via `ManualClipIcon` on each gallery grid thumbnail, so the gallery's "origin indicator" requirement is met honestly (there is exactly one possible origin right now) and the type only needs to grow, not be reworked, if a second trigger is ever reintroduced.
- **Real clip duration on a cold directory listing** (`listSavedClips()` in `utils/files.ts`, used on Gallery mount and after every app restart) has the same problem the stale entry above was trying to solve for `triggeredBy`: `SaveClipResult.durationSeconds` only exists in memory for the JS session that just recorded the clip, and decoding the video file to recover it on every gallery render is exactly what the "don't re-decode video" performance requirement rules out. So duration *is* encoded in the filename this time, for real: `useCircularBuffer.stopRecording()` renames the just-saved file via `RNFS.moveFile` to `clip_<epochMillis>_d<durationMs>.mp4` (`encodeClipFilename`/`CLIP_FILENAME_PATTERN` in `utils/files.ts`), and `listSavedClips()` parses the `_d<ms>` suffix back out, falling back to `0` for any pre-existing clip saved before this change (whose filename has no suffix). If the rename itself fails (rare, e.g. a concurrent filesystem issue), `stopRecording()` keeps the clip under its original un-renamed path rather than failing the whole save — it just won't show a real duration until re-derived some other way.

## Gallery rebuild (grid, preview, multi-select, delete confirmation)

- **Grid layout**: a single virtualized `FlatList`, not `SectionList` + `numColumns`. `SectionList` doesn't support a `numColumns` grid directly, and `FlatList`'s own `numColumns` can't mix full-width date headers into an otherwise fixed-column grid in the same list. Instead, `buildGalleryListItems` (`utils/dateGroups.ts`) pre-flattens clips into a single array of `{type:'header'}` / `{type:'row', clips: SavedClip[]}` items (rows pre-chunked to `numColumns` clips each), and `GalleryScreen` renders that with one `FlatList` — still fully virtualized (only visible rows mount), just without relying on `numColumns` to do the chunking.
- **Responsive columns**: 2 columns by default, 3 when `useWindowDimensions().width >= 700` (covers landscape phones and tablets). No separate breakpoint config — picked one threshold rather than a multi-tier system, since the brief only asked for "2–3 conforme largura."
- **Date grouping**: `dateGroupTitle` (`utils/dateGroups.ts`) returns "Hoje"/"Ontem" for the last two calendar days (compared via local `Date`, not UTC-normalized — matches how a user reads "today" on their own device) and `"<day> de <mês>"` (PT-BR month names, hand-rolled array rather than `Intl.DateTimeFormat` to avoid depending on full ICU data being present in the Hermes build) otherwise.
- **Thumbnail caching**: `getClipThumbnail` (`utils/thumbnails.ts`) now memoizes by clip path in a module-level `Map<string, Promise<string|null>>`, so scrolling a thumbnail out of the `FlatList`'s virtualization window and back doesn't re-invoke `react-native-create-thumbnail` (a native frame-decode) a second time. Safe to cache indefinitely because clip filenames are unique/immutable (the timestamp is baked in) — a given path is always the same video. A failed lookup is *not* cached (so a transient failure, e.g. the file not fully flushed yet, can be retried), and `deleteClip` evicts the cache entry for a path it removes.
- **Multi-select**: long-press (`onLongPress` on `ClipGridThumbnail`) enters selection mode by selecting that one clip; subsequent taps toggle membership. The "X selecionados" counter and a "Cancelar" button live in the *native stack header* (via `navigation.setOptions`), matching where iOS Photos/Google Photos put it — not in the bottom toolbar, which only has the share/delete icons. `no-unstable-nested-components` (eslint) flagged the first version of the `headerLeft` factory closing over JSX inline; fixed by pulling the button into a real top-level `SelectionCancelButton` component and passing a `useCallback`-memoized factory, rather than defining a fresh component function on every render.
- **Delete confirmation is a single shared `DeleteConfirmModal`**, used identically from `GalleryScreen` (bulk, via the selection toolbar) and `ClipPreviewScreen` (single clip) — one component, one place that could get the "must confirm before deleting" rule wrong, not two independent implementations that could drift.
- **Bulk share** uses `Share.open({ urls: [...] })` (plural) when more than one clip is selected, `Share.open({ url })` (singular) for exactly one — `react-native-share`'s API distinguishes the two; sharing a single-element `urls` array works on both platforms but the singular form is what the rest of the app's single-clip share already used, so it stays consistent for the 1-clip case.
- **Full-screen preview player**: added **`react-native-video`** (the brief's own suggestion, and the standard, well-supported choice — confirmed it links and builds clean via `pod install` + a full `xcodebuild` simulator build, and `./gradlew :app:compileDebugKotlin` on Android). Controls (`ClipPreviewScreen`) reuse `@react-native-community/slider` (already a dependency, used for the buffer-seconds setting) for the scrub bar rather than adding a second slider library. Tapping the video toggles play/pause; a centered play glyph shows only while paused (matches the standard video-app convention of not covering the frame with a permanent button).
- **"Open in system photos app" button was removed from the preview screen after physical-device testing** — it originally reused the exact same `Share.open({ ..., showAppsToView: true })` approach `GalleryScreen` used before this rebuild, on the theory that `showAppsToView` biases the sheet toward viewer apps. Running on a real iPhone showed "Compartilhar" and "Abrir no app de fotos" opening the identical share sheet with no visible difference — checking `react-native-share`'s source confirmed why: `showAppsToView` has **zero references anywhere in the iOS native implementation** (`ios/RNShare/`), it's an Android-only option (`android/.../ShareIntent.java`). So on iOS the two buttons were calling literally the same code with different-looking icons. Removed the redundant button/icon (`ExternalOpenIcon`, `handleOpen`) rather than keep dead UI around; the preview screen's top bar is now just Back / Share / Delete. If a real "open in Photos" distinction is wanted on Android specifically (where `showAppsToView` does do something), that would need to be a platform-conditional feature added deliberately, not a copy-pasted no-op.

## Camera zoom

- **No `react-native-vision-camera` in this project** (the brief's own suggestion assumed it was already in use) — `CircularBufferPreview` is a fully custom native camera pipeline (`AVCaptureSession` on iOS, Camera2 on Android, see `CameraEncoderController.swift`/`.kt`), built that way because the ring-buffer encoder needs to observe the exact same frames the preview shows. So zoom had to be added to *that* custom pipeline directly, in native code on both platforms — there was no higher-level camera library's zoom API available to lean on here, unlike what the brief anticipated.
- **iOS**: camera device selection changed from a fixed `AVCaptureDevice.default(.builtInWideAngleCamera, ...)` to `pickBackCameraDevice()`, which prefers a virtual multi-lens device (`.builtInTripleCamera`, then `.builtInDualWideCamera`, falling back to the original single-lens `.builtInWideAngleCamera` on older/cheaper devices). A virtual multi-lens device's `videoZoomFactor` sweeps continuously across its physical lenses — including below 1.0 for ultra-wide — with the OS handling the lens handoff, exactly like the stock Camera app; `minAvailableVideoZoomFactor`/`maxAvailableVideoZoomFactor` report the real range (min < 1.0 only when ultra-wide is actually in the set), which is what `hasUltraWide` is derived from (not the device type directly, so it stays correct even if some future Apple device's zoom-1.0 reference lens isn't the wide-angle component).
- **Android**: zoom uses `CaptureRequest.CONTROL_ZOOM_RATIO` (API 30+), which the framework maps across a logical multi-camera's physical lenses automatically, including below 1.0 for ultra-wide, read from `CameraCharacteristics.CONTROL_ZOOM_RATIO_RANGE`. Below API 30 (this app's `minSdkVersion` is 24) or on a device that simply doesn't report a zoom-ratio range, it falls back to manual `SCALER_CROP_REGION` digital zoom (`cropRegionForZoom`) clamped to `1.0..SCALER_AVAILABLE_MAX_DIGITAL_ZOOM` — crop-region zoom can never go below 1.0, so `hasUltraWide` correctly comes back `false` in that fallback path rather than offering a 0.5x pill that wouldn't do anything.
- **Zoom cannot interfere with the running buffer, on either platform, by construction, not just by testing**: iOS's `setZoom` only calls `device.lockForConfiguration()` + sets `videoZoomFactor` — it never calls `session.beginConfiguration()`/`commitConfiguration()` or touches inputs/outputs, so it can't restart the `AVCaptureSession`. Android's `setZoom` mutates the *existing* `CaptureRequest.Builder` (promoted from a local variable in `createCaptureSession` to a stored field for exactly this reason) and resubmits it via `session.setRepeatingRequest(...)` on the same already-open `CameraCaptureSession` — it never calls `createCaptureSession` again, so it can't restart the session either. Verified by reading through both encoder controllers' full lifecycle (there's no code path from `setZoom` back into camera-open/session-recreate on either platform) and by a full `xcodebuild`/`gradlew compileDebugKotlin` build on both platforms; **not verified against a physical device with the buffer actively recording while zooming**, since this sandboxed environment has no attached hardware and the camera doesn't work in the iOS Simulator (see README "Known limitations") — this is the one thing from the Definition of Done that still needs a manual pass on a real device before shipping.
- **Native zoom-range query is polled briefly, not queried once**: `useZoom.ts` retries `getZoomInfo()` every 300ms (up to ~3.6s) after the buffer reaches `'buffering'`, because — per the existing "buffer never actually started" bug entry above — `startBuffering()`'s promise resolving doesn't mean the camera device has actually finished opening on the native side yet. A single immediate query could plausibly race that and permanently fail to find a device, leaving the zoom pills stuck hidden for the rest of the session.
- **Pinch-to-zoom via core `PanResponder`**, not `react-native-gesture-handler`: multi-touch is available on `evt.nativeEvent.touches` without an extra library, consistent with this project's established preference for avoiding new native dependencies when a built-in API already covers the need (see the blur-vs-scrim decision above). The actual pinch math (`PinchZoomTracker`, `utils/pinchZoom.ts`) is deliberately kept independent of `PanResponder` itself, because `PanResponder`'s real responder-system event objects carry an internal `touchHistory` that a hand-built fake event in a test can't reproduce (confirmed by trying — calling the wrapped `onPanResponderGrant`/`onPanResponderMove` handlers directly against a plain mock event crashes inside React Native's own `TouchHistoryMath`). Testing the pure tracker class directly instead is the same "test the pure algorithm, not the hardware/framework-bound wrapper" pattern already used for `SaveClipDurationTest`/`SaveClipDurationTests`.
- **Zoom level persists only for the current session** (`useCameraStore`, zustand, in-memory), not across app restarts — consistent with `settingsStore`'s existing behavior (settings already reset every launch, a known pre-existing simplification, see "Onboarding" above) and with what was actually asked for ("não precisa persistir entre sessões diferentes").
- **0.5x-unavailable behavior: hidden, not shown-disabled.** `computeZoomPillLevels` simply omits the 0.5x pill when `hasUltraWide` is false, per the brief's own stated default. Confirmed with the user (asked per the brief's explicit request) — hidden is correct.

### Camera zoom — 0.5x root cause and permanent fix (physical-device testing, 2026-07-13)

**The claim above that ultra-wide (sub-1.0) zoom "just works" via `videoZoomFactor` on a virtual multi-lens `AVCaptureDevice`, using a plain threshold check (`hasUltraWide = minZoom <= 0.6`), was wrong** — not because of a session/architecture limitation (an earlier version of this entry concluded `AVCaptureMultiCamSession` was required; that conclusion was itself wrong and is superseded by this one), but because of a wrong mental model of what AVFoundation's `videoZoomFactor` actually measures on a virtual multi-lens device. Found by testing on the user's physical iPhone (a triple-camera model), then confirmed against an Apple DTS engineer's own worked example on the Apple Developer Forums (thread 719602) before implementing the fix — not guessed:

- **The real semantics**: `videoZoomFactor` is a **device-relative** internal unit, not the "x" number the stock Camera app displays. On a virtual triple/dual-wide-camera device, `1.0` is simply *that device's own minimum* — which, because the ultra-wide lens is the widest constituent, **is the ultra-wide lens**, not the wide lens. The wide lens (the real "1x" reference point) is `device.virtualDeviceSwitchOverVideoZoomFactors.first`. Confirmed on the user's device: `minAvailableVideoZoomFactor == 1.0` (ultra-wide), `virtualDeviceSwitchOverVideoZoomFactors == [2, 8]` (`2.0` = wide lens/"1x", `8.0` = telephoto/"4x") — the exact same pattern the Apple engineer's iPhone 14 Pro example describes (`[2, 6]` there instead of `[2, 8]`, different tele reach, same structure). The old `minZoom <= 0.6` check was checking whether the device's *own* minimum was below an arbitrary threshold, which is never true for *any* virtual multi-lens device (its own minimum is always `1.0`, whether or not that `1.0` happens to be ultra-wide) — so 0.5x was unreachable on literally every triple/dual-wide-camera iPhone, not just this one.
- **The permanent fix** (`CameraEncoderController.swift`): a new `wideLensReferenceZoom(for:)` helper returns `virtualDeviceSwitchOverVideoZoomFactors.first` (falling back to `minAvailableVideoZoomFactor` itself for a plain single-lens device with no switchover array, where display and internal units are already the same). `zoomInfo()` and `setZoom(factor:)` now convert between AVFoundation's internal units and **display** units (`display = internal / referenceZoom`, and the inverse for applying a requested display zoom) before returning/applying anything; `hasUltraWide` is now `minAvailableVideoZoomFactor < referenceZoom` (is there real range *below* the wide-lens reference), not a hardcoded threshold. No JS changes were needed — `useZoom.ts`/`computeZoomPillLevels` already treated whatever the native side reported as display units; the bug was entirely in what the native side was reporting.
- **No architecture change needed.** This is a pure unit-conversion fix confined to the existing single-stream `AVCaptureSession` (`CameraEncoderController`'s `session`/`AVCaptureVideoDataOutput` pipeline is untouched) — `AVCaptureMultiCamSession` was never actually required; that theory came from an incomplete first read of the symptom (forcing `videoZoomFactor = 0.5` directly crashes with `NSRangeException` on the *unconverted* internal scale, since `0.5` is below `minAvailableVideoZoomFactor` on this device's own units — it doesn't mean the range is unreachable, it means `0.5` needs to be requested in internal units as `0.5 * referenceZoom`, i.e. `1.0`, which is exactly the device's real minimum).
- **Verified end-to-end on the physical device**: a temporary read-only debug print (no forced/invalid property writes this time) confirmed `zoomInfo()` now returns `min=0.5, max=61.875, hasUltraWide=true` on the user's phone — matching the real hardware exactly. Removed before shipping, per the same debug-instrumentation-cleanup discipline as every other physical-device bug hunt in this doc.
- **Android was not re-verified against a physical device with an ultra-wide lens in this pass** (no Android hardware was available at the time). Android's `CONTROL_ZOOM_RATIO` is documented by Google as already being display-relative (a ratio of `0.5` is explicitly "the ultra-wide camera relative to the wide camera", not a device-internal unit needing translation the way iOS's does), so the existing Android implementation is not expected to have the same bug — but "not expected to" isn't "confirmed," and this should still get a real device check before being trusted at face value.
- **Follow-up bug from this same fix, found immediately on the next physical-device check**: the camera opened with the sensor actually framed at 0.5x (correct, now that the unit conversion is right — a fresh `AVCaptureDevice` defaults to its own internal minimum, which is display `0.5x` on this device), but the zoom pills still showed "1x" highlighted, because nothing had ever told the hardware to *go to* display `1x` — the JS side only calls `setZoom()` in response to a pill tap or a pinch, never on initial camera open. Fixed in `useZoom.ts`: right after a successful `getZoomInfo()` (i.e. every time the camera opens or reopens — this hook's polling effect runs on every fresh mount, not just app first-launch), it now reads the current target zoom from `useCameraStore.getState().zoomFactor` (the store's default of `1` on a truly fresh session, or the user's last-picked level if the camera is reopening after navigating to Settings/Gallery and back within the same session — the native `AVCaptureSession` is fully torn down and reopened on every such reattach, so the hardware needs re-syncing every time, not just once) and immediately calls `CircularBufferModule.setZoom()` with it, so the actual lens always matches whatever the UI displays as current, rather than only self-correcting the next time the user interacts with zoom.

## iOS build

- `IPHONEOS_DEPLOYMENT_TARGET` raised from 15.1 to 16.0 (project.pbxproj, all 4 build configs) and the `Podfile`'s `platform :ios, min_ios_version_supported` pinned to a literal `'16.0'` instead of react-native's own 15.1 floor. Required because `porcupine-react-native`'s podspec declares `s.platforms = { :ios => "16.0" }`; this is also the minimum for `AppIntents`/`AppShortcutsProvider` (Section D), which is a convenient coincidence rather than the reason for the bump.
- `@picovoice/react-native-voice-processor@1.2.3`'s podspec declares `RCT-Folly`/`React-Codegen`/`RCTRequired`/`RCTTypeSafety`/`ReactCommon/turbomodule/core` as dependencies under `RCT_NEW_ARCH_ENABLED=1`, but RN 0.86 ships those bundled inside a prebuilt `ReactNativeCore`/`ReactNativeDependencies` tarball and no longer publishes them as standalone pods, so `pod install` failed with "Unable to find a specification for RCT-Folly". Neither `VoiceProcessor.swift` nor `VoiceProcessor.m` actually reference Folly/Codegen/turbomodule symbols — that dependency block was defensive boilerplate from an older new-arch template, not a real compile-time requirement — so it's safe to drop. Patched via `patch-package` (`patches/@picovoice+react-native-voice-processor+1.2.3.patch`, applied automatically by the `postinstall` script) rather than hand-editing `node_modules`, so the fix survives `yarn install`.
- The `CircularBuffer` Xcode group had `path = "../CircularBuffer"` instead of `"CircularBuffer"`, resolving one directory above `ios/` (i.e. to the repo root instead of `ios/CircularBuffer/`). Every build before this fix would have failed with "Build input files cannot be found" for all seven native buffer files — meaning the iOS app had apparently never been compiled before this audit. Fixed via the `xcodeproj` Ruby gem (`group.path = "CircularBuffer"`), not by hand-editing `project.pbxproj`.
- `CircularBufferPreviewView.isActive` was a plain (non-`@objc`) Swift property. `RCT_EXPORT_VIEW_PROPERTY(isActive, BOOL)` bridges props via Objective-C KVC (`-setIsActive:`), and Swift 4+ does not implicitly expose non-`@objc` members to the ObjC runtime even in an `NSObject`/`UIView` subclass. The very first prop update after the view mounted crashed with `unrecognized selector sent to instance` — this is the core camera-preview integration, so this crashed on every real launch. Fixed by marking it `@objc dynamic var isActive: Bool`. Found only by actually running the app (via Detox), not by static review — see `AUDIT_REPORT.md`.
- `react-native-permissions` ships with **zero permission handlers compiled in** unless the Podfile calls `setup_permissions([...])` (it rewrites `RNPermissions.podspec`'s `source_files`/`frameworks` at `pod install` time to include only the requested permission subspecs). Without it, `check()`/`request()` fail at runtime with "No permission handler detected", which `usePermissions.ts` doesn't handle as a distinct case (it would just look like every permission is stuck `unknown`/`denied` forever). Added `setup_permissions(['Camera', 'Microphone', 'PhotoLibraryAddOnly'])` to the Podfile, matching the three usage descriptions in `Info.plist`.

- **`DEVELOPMENT_TEAM` set to `UZBGRC8T87`** (Automatic signing) on the `VoiceCamBuffer` target, so the project can build and run on a physical device. This is a personal Apple ID's free provisioning team on this specific development machine, picked by the user from 3 signed-in Xcode account sessions (a distinct company team, "Field Vision Sports LLC", was also available and explicitly not chosen). If this project ever moves to a different machine or a paid company team for App Store distribution, this value will need to change.

## Native error event plumbing

- `CircularBufferModule` on both platforms could only reject the `startBuffering()` promise for errors that happen *before* it resolves; camera-busy/session-interrupted errors that occur afterward (the promise already resolved once the encoder/session setup was merely *kicked off*) had no pending Promise to reject and were silently `Log.e`/`NSLog`'d only — the JS side never found out, so the UI stayed on "buffering" forever with no error banner for a failure that had already happened. Fixed by turning `CircularBufferModule` into a real event emitter (Android: `DeviceEventManagerModule.RCTDeviceEventEmitter` + no-op `addListener`/`removeListeners`; iOS: subclass `RCTEventEmitter`, override `supportedEvents()`/`moduleName()`/`requiresMainQueueSetup()` with `override`) sending a `"CircularBufferError"` event, which `useCircularBuffer.ts` now subscribes to via the already-scaffolded (but previously unused) `circularBufferEvents` export and routes into the same `setError()` path as a rejected `saveClip()`.

## Testing infrastructure

- Nothing under `jest`, `@testing-library/react-native`, or Detox had ever actually been run in this repo before this audit — `@react-native/jest-preset` and the `test-renderer` peer dependency (a *different* package from `react-test-renderer`, required by `@testing-library/react-native@14`'s `renderHook`/`render`) were both missing from `devDependencies`; `jest.config.js` had no `transformIgnorePatterns` for ESM-only RN libraries and no native-module mocks (`jest.setup.js` + `__mocks__/react-native-config.js` now cover `react-native-permissions`, `@react-native-documents/picker`, `react-native-share`, `react-native-fs`, both `@picovoice/*` packages, and `react-native-config`).
- `@testing-library/react-native@14`'s `render`/`renderHook` are **async** (must be `await`ed) and its custom `test-renderer` needs `await act(async () => ...)` — the sync `act(() => ...)` form silently fails to flush state updates scheduled outside React's own render call stack (e.g. a Porcupine detection callback firing from a mocked native module). Every test in this repo uses the async forms for this reason.
- `jest.config.js`'s `testPathIgnorePatterns` excludes `e2e/` — that directory is a separate Detox test runner (its own `e2e/jest.config.js`, invoked via `detox test`, not plain `jest`), and picking it up under the regular `yarn test` run fails immediately since the Detox global environment isn't present outside that runner.
- The iOS `VoiceCamBufferTests` XCTest target is a **standalone logic-only test bundle** (no `TEST_HOST`/host-app launch) — `FrameRingBuffer.swift`/`EncodedFrame.swift` have zero AVFoundation/UIKit dependency, so they're added directly to the test target's own Sources build phase (same source files, compiled into both binaries independently) rather than using `@testable import VoiceCamBuffer`, mirroring the Android side's plain-JUnit `FrameRingBufferTest.kt` (no Robolectric/instrumentation needed either). Created and wired into the shared scheme's Test action via the `xcodeproj` Ruby gem.
- Detox is configured (`.detoxrc.js`, `e2e/jest.config.js`, `e2e/smoke.test.ts`) and the full pipeline (build → install → launch → connect → grant permissions) was verified working end-to-end against a real simulator, catching the two real native bugs documented above in the process. The test assertions themselves are currently blocked by a Detox-vs-RN synchronization quirk unrelated to app correctness — see `BLOCKED.md`.
